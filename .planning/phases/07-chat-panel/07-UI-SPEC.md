# Phase 7 -- UI Design Contract

> Visual and interaction contract for the per-workbench chat panel. Generated by gsd-ui-researcher.
>
> This phase replaces `ChatPlaceholder.tsx` with a full chat panel in the right column of AppShell. Each workbench has its own independent chat backed by its persistent Claude session (Phase 6). Agent messages render as rich markdown with streaming. Tool calls display as collapsible cards. AskUserQuestion tool calls render as interactive UI components from the design system. A fixed input row stays at the bottom. Session history is accessible via an expandable menu. All visual values inherit from the Phase 3 design system (theme.css tokens, component library).

## Spacing Scale

Inherited from Phase 3 (03-UI-SPEC.md). Chat-specific spacing applications:

| Context | Token | Value | Rationale |
|---------|-------|-------|-----------|
| Chat panel internal padding | p-0 | 0px | Full-bleed layout -- message list fills the panel edge-to-edge horizontally |
| Message list vertical padding | py-4 | 16px | Standard breathing room at top and bottom of message list |
| Message bubble horizontal padding | px-4 | 16px | Standard element spacing for readable text |
| Message bubble vertical padding | py-3 | 12px | 12px exception: compact vertical rhythm for conversational messages without excessive whitespace between lines |
| Message gap (between messages) | gap-4 | 16px | Clear separation between user and agent messages |
| Input row padding | p-3 | 12px | 12px exception: compact input area preserves vertical space for message history |
| Input row gap (input to button) | gap-2 | 8px | Tight coupling of input field and submit button |
| Tool card internal padding | p-3 | 12px | 12px exception: compact card content for collapsible sections |
| Tool card header padding | px-3 py-2 | 12px / 8px | Compact header for tool cards within message flow |
| AskUserQuestion option gap | gap-3 | 12px | 12px exception: spacing between radio/checkbox options in interactive tool calls |
| Chat header height | h-10 | 40px | Consistent with other panel headers in the app |
| Chat header padding | px-3 | 12px | Matches status panel header pattern (Phase 6) |
| Session history menu item padding | px-3 py-2 | 12px / 8px | Compact menu items for session list |
| Code block padding | p-3 | 12px | 12px exception: readable code block content without excessive margins |

Exceptions: 12px (p-3, px-3, py-3, gap-3) is used throughout this phase. Justification: chat panels are conversational, high-density UI where 16px creates excessive whitespace between messages and controls, while 8px is too cramped for readability. This matches the Phase 6 precedent for compact secondary panels.

| Property | Value |
|----------|-------|
| Strategy | Token-by-token append to current message content. Re-render markdown on each chunk. |

| Buffer | No buffering -- render each token as it arrives for maximum responsiveness. |

## AskUserQuestion Contract (CONTEXT.md D-04, D-05)

### Adapter Layer (Pitfall P15 mitigation)

Parse the AskUserQuestion tool call JSON into an intermediate format before rendering. This isolates the UI from Agent SDK schema changes.

```typescript
interface ParsedQuestion {
  type: 'single_select' | 'multi_select' | 'free_text' | 'confirmation';
  question: string;
  options?: Array<{
    value: string;
    label: string;
    description?: string;
  }>;
  defaultValue?: string | string[];
}
```

If the AskUserQuestion schema is unrecognized, fall back to rendering as plain text within a standard agent message (not a crash, not a blank space).

### Rendering Per Type

| Question Type | UI Component | Design System Components Used |
|---------------|-------------|-------------------------------|
| single_select | Radio button group with option cards | RadioGroup, RadioGroupItem, Card (flat variant) |
| multi_select | Checkbox group with option cards | Checkbox, Card (flat variant) |
| free_text | Text input with submit | Input (md size), Button (primary variant, sm size) |
| confirmation | Two-button prompt | Button (primary + ghost variants) |

### Option Card Layout

Each option in single_select and multi_select renders as:

| Property | Value |
|----------|-------|
| Container | Card, flat variant, interactive=true. 100% width. |
| Content | Left: RadioGroupItem or Checkbox (vertically centered). Right: label (text-body) + description (text-caption text-text-dim, optional). |
| Selected state | Border becomes accent. Background becomes surface-3. |
| Padding | p-3 |
| Gap between selector and text | gap-3 (12px) |
| Gap between options | gap-2 (8px) |

### Submit Behavior

| Property | Value |
|----------|-------|
| Submit button | "Send Response" label, primary variant, sm size. Positioned below options. |
| Disabled state | Disabled until user makes a selection (single_select/multi_select) or enters text (free_text). |
| After submission | Options become read-only (disabled state). Selected option highlighted. A small "Submitted" badge (success variant, sm) appears. |
| Response routing | Captured selection is sent back to the backend via WebSocket as a tool response for the pending AskUserQuestion call. |

## Chat Store Contract (Zustand)

New `chatStore.ts` managing per-workbench chat state:

```typescript
type MessageRole = 'user' | 'agent' | 'system';
type MessageStatus = 'sending' | 'sent' | 'streaming' | 'complete' | 'error';

interface ToolCall {
  id: string;
  name: string;
  input: string;       // JSON string
  output: string | null;
  status: 'running' | 'complete' | 'error';
}

interface AskUserQuestion {
  toolCallId: string;
  question: ParsedQuestion;   // Adapter layer output
  response: string | null;    // User's response once submitted
  submitted: boolean;
}

interface ChatMessage {
  id: string;
  role: MessageRole;
  content: string;
  timestamp: string;          // ISO timestamp
  status: MessageStatus;
  toolCalls?: ToolCall[];
  askUserQuestion?: AskUserQuestion;
}

interface WorkbenchChat {
  workbenchId: WorkbenchId;
  messages: ChatMessage[];
  inputValue: string;
  isStreaming: boolean;
  activeSessionId: string | null;
}

interface ChatState {
  chats: Map<WorkbenchId, WorkbenchChat>;
  getChat: (id: WorkbenchId) => WorkbenchChat;
  addMessage: (id: WorkbenchId, message: ChatMessage) => void;
  updateMessage: (id: WorkbenchId, messageId: string, update: Partial<ChatMessage>) => void;
  appendStreamChunk: (id: WorkbenchId, messageId: string, chunk: string) => void;
  setInputValue: (id: WorkbenchId, value: string) => void;
  setStreaming: (id: WorkbenchId, streaming: boolean) => void;
  updateToolCall: (id: WorkbenchId, messageId: string, toolCallId: string, update: Partial<ToolCall>) => void;
  submitAskUserResponse: (id: WorkbenchId, messageId: string, response: string) => void;
  loadSessionHistory: (id: WorkbenchId, messages: ChatMessage[]) => void;
  clearChat: (id: WorkbenchId) => void;
}
```

### WebSocket Event Mapping

| WebSocket Event | Store Action |
|----------------|--------------|
| chat:message (role=agent, streaming start) | addMessage with status "streaming", setStreaming true |
| chat:stream-chunk | appendStreamChunk to current streaming message |
| chat:stream-end | updateMessage status to "complete", setStreaming false |
| chat:tool-call-start | Add ToolCall to current agent message, status "running" |
| chat:tool-call-end | Update ToolCall status to "complete" or "error", set output |
| chat:ask-user | Add AskUserQuestion to current agent message |
| chat:error | addMessage with role "system", status "error" |
| session:connected | Update activeSessionId for workbench |
| session:disconnected | Set activeSessionId to null |

## Accessibility Contract

| Requirement | Implementation |
|-------------|----------------|
| Focus visible | All interactive elements receive 2px solid accent outline + glow on keyboard focus (inherited from globals.css) |
| Keyboard navigation | Tab moves through header buttons, message list, interactive tool calls, input, and send button. Arrow keys navigate radio groups. |
| Screen reader | Chat panel has role="log" with aria-label="Chat with {Workbench} agent". Messages use role="listitem" semantics within a role="list" container. User messages have aria-label="You said: {content preview}". Agent messages have aria-label="{Workbench} agent said: {content preview}". |
| Tool cards | role="button" with aria-expanded on collapsed/expanded toggle. aria-label="{ToolName} tool call -- click to expand". |
| AskUserQuestion | RadioGroup and Checkbox use built-in Radix ARIA. Question text is a visible label with htmlFor linkage. Submit button aria-label="Send response". |
| Input | Textarea has aria-label="Message input". Send button has aria-label="Send message". |
| Overflow menu button | aria-label="Chat panel options". Tooltip: "More options". |
| Streaming | aria-live="polite" on the streaming message container. Screen reader announces "Agent is responding" at stream start and "Agent response complete" at stream end. |
| Reduced motion | Streaming cursor blink disabled. Message appear animations disabled. Tool card expand animations disabled. |
| Touch targets | All buttons meet 32px minimum (icon button = 40px, send button = 40px height). |
| Color independence | Message role communicated through position (left/right), shape (bubble vs full-width), AND screen reader labels -- not color alone. Tool status communicated through icon shape AND text AND color. |
| Live region | Scroll-to-bottom button has aria-label="Scroll to bottom, {N} new messages" (count updated dynamically). |

## Performance Contract

| Concern | Mitigation |
|---------|-----------|
| Re-render isolation | Chat state in dedicated chatStore. MessageBubble components use React.memo with message.id + message.status as deps. Only the streaming message re-renders on chunk append. |
| Long message lists | react-window virtualization for message lists exceeding 100 messages. Below 100: standard DOM rendering. Threshold chosen because chat messages are variable-height. |
| Streaming re-renders | appendStreamChunk updates a single message's content string. Markdown re-parse on each chunk is acceptable (react-markdown is fast for small incremental changes). If performance issues arise: batch chunks on 50ms RAF timer. |
| Syntax highlighting | shiki highlight is async. Code blocks render with bg-surface-2 placeholder, then syntax-highlighted version swaps in. Use shiki's `codeToHtml` with caching (same code + language = cached result). |
| Tool card expanded content | Lazy-render expanded content only when tool card is expanded (not on mount). |
| Session history loading | Loading indicator while session messages fetch. Messages load in a single batch, not streamed. |
| Image/attachment rendering | Out of scope for Phase 7. No image rendering in messages. |
