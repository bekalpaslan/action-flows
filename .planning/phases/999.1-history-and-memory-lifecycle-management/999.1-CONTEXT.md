# Phase 999.1: History & Memory Lifecycle Management - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Implement a 3-layer history model (raw → ledger → canonical) to prevent unbounded growth of three data stores:
- `packages/backend/data/gate-traces.jsonl` — 520 lines, 380KB, grows with every gate evaluation
- `.claude/actionflows/logs/` — 14MB, 40+ log session directories, inconsistent INDEX.md coverage
- `.claude/actionflows/LEARNINGS.md` — 261 lines, no cap or archiving

Add TTL for raw operational data, create a searchable execution ledger, establish promotion pipelines (raw→ledger, ledger→canonical), and enforce INDEX.md creation for new log session directories.

New capabilities (search UI, bulk review, log visualization) are separate phases.

</domain>

<decisions>
## Implementation Decisions

### Raw data TTL
- **D-01:** Gate traces pruned after **7 days** — matches existing CleanupService retention policy
- **D-02:** CleanupService extended to cover `gate-traces.jsonl`, not just file-persistence date folders

### Ledger structure
- **D-03:** Structured summary ledger entry fields: `timestamp`, `sessionId`, `gate/phase name`, `pass/fail outcome`, `health score delta`, `short reason string`
- **D-04:** Ledger lives in `packages/backend/data/ledger.jsonl` alongside gate-traces
- **D-05:** Backend exposes `GET /api/history/ledger` for frontend queries (returns ledger entries with optional filters)

### Promotion pipeline — raw → ledger
- **D-06:** Auto-promote at TTL expiry: when a raw gate trace approaches 7-day limit, summarize it into a ledger entry before pruning. No user action required.

### Promotion pipeline — ledger → canonical
- **D-07:** Healing approval in LearningsBrowser triggers promotion to LEARNINGS.md. This is the natural flow already established in Phase 10 (approved attempts as provisional learnings). No additional trigger needed.

### LEARNINGS.md lifecycle
- **D-08:** Cap active LEARNINGS.md at **50 entries**
- **D-09:** Entries beyond cap are archived to `LEARNINGS-archive-YYYY.md` (one file per year). Active file stays fast to load; history is not lost.
- **D-10:** Archive file is appended to, not replaced — ordering is oldest-first within each year file.

### logs/INDEX.md trust gap
- **D-11:** Enforce INDEX.md creation at log session dir creation time — middleware or log-init hook creates it when a new session directory is made. Old dirs without INDEX.md are accepted as legacy (no backfill).
- **D-12:** INDEX.md minimum content: session dir name, action type (analyze/code/plan/etc.), timestamp, and a one-line description (auto-generated from the task input).

### Claude's Discretion
- Exact prune scheduling strategy (run inline with CleanupService vs separate cron)
- How to handle gate traces with missing fields during promotion
- Ledger file rotation if ledger.jsonl itself grows large (out of scope for v1, Claude decides)
- Index entry format details

</decisions>

<specifics>
## Specific Ideas

- The 3-layer model mirrors established data lifecycle patterns (operational → analytical → canonical)
- Phase 10 `LearningsBrowser` shows approved healing attempts as provisional learnings — this phase formalizes the promotion step so approved attempts actually land in LEARNINGS.md
- "Cap + archive" approach is deliberately conservative: 50 entries keeps active file agent-readable in a single context load

</specifics>
