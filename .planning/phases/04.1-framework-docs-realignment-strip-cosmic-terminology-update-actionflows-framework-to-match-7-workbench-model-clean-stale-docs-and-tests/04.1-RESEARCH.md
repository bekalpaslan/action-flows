## Summary

This phase is a large-scale terminology replacement and structural realignment of the `.claude/actionflows/` framework files, plus deletion of the top-level `docs/` folder and all historical logs. The actual file count is 537 .md files in actionflows (not 941 as originally estimated in CONTEXT.md), of which 385 are log files being deleted wholesale. The remaining 152 non-log .md files break into tiers: ~15 critical instruction files needing deep rewrite, ~30 files needing moderate terminology replacement, ~50+ files needing light search-replace or no changes, and 5 internal docs files (some heavily cosmic, some to delete). Additionally, 108 files in the top-level `docs/` folder will be deleted.

The key complexity is NOT the volume -- it is the semantic nuance of the "gate" and "harmony" terms. "Gate" is used both as a cosmic metaphor (Gates = checkpoints in space) AND as functional orchestrator vocabulary (Pre-Action Gate, Gate 1, Gate 2, etc.). "Harmony" is both cosmic (The Void/Harmony) AND a real system concept (harmony detection, harmony health scores). The terminology mapping from D-06 must be applied carefully: cosmic uses of "gate" become "validation checkpoint" while functional uses of "Pre-Action Gate" stay as-is. Cosmic uses of "harmony" become "system health" while the technical harmony detection system retains its name (or transitions to "health" consistently).

**Primary recommendation:** Delete first (docs/, logs/), then deep-rewrite critical files (ORCHESTRATOR.md, CONTEXTS.md, project.config.md, CLAUDE.md), then batch search-replace remaining files. Order of operations matters because deep-rewrite files establish the new vocabulary that all other files must match.

## Architecture Patterns

### Terminology Overlap Analysis (CRITICAL)

The most dangerous aspect of this phase is that several "cosmic" terms double as functional vocabulary. The planner MUST distinguish these:

#### "Gate" -- Dual Usage
- **Cosmic usage** (replace): "Gates are checkpoints in space", "Stars signal through gates", `GateCheckpoint` cosmic framing
- **Functional usage** (KEEP): "Pre-Action Gate", "Gate 1", "Gate 2", "Gate 4 violations", `gateCheckpoint.ts`, `gateTraces`, gate logging
- **Rule:** The word "gate" in ORCHESTRATOR.md's "Pre-Action Gate" section (Gates 1-14) is functional orchestrator vocabulary. Do NOT rename these. Only rename cosmic descriptions of what gates ARE (e.g., "checkpoints in space" -> "validation checkpoints").
- **D-06 says:** Gates -> Validation Layer (neural validation). This applies to the COSMIC CONCEPT of gates (the entity type), not every instance of the word "gate" in the codebase.

#### "Harmony" -- Dual Usage
- **Cosmic usage** (replace): "The Void/Harmony", "harmony is the space between workbenches", "the terrain", "cosmic background"
- **System usage** (rename carefully): `harmony` context in CONTEXTS.md, harmony detection, harmony health scores, `harmonyDetector.ts`, `harmony:check`
- **Rule:** The `harmony` auto-target context becomes `system-health` (or similar). References to the harmony detection SYSTEM should be updated to "system health" consistently. Code references (`harmonyDetector.ts`, API paths `/api/harmony/health`) are OUT OF SCOPE for this phase (code changes are separate).
- **Preserve:** CONTRACT.md references to harmony detection as a technical system stay but get terminology updates in the prose.

#### "Spark" -- Clean Replace
- Only used in cosmic model docs and SYSTEM.md. Replace with "Agent" everywhere.

#### "Star" -- Clean Replace with Caution
- Cosmic "Stars = Workbenches" is clear replace territory.
- WATCH OUT for: "start" (common word), "restart", "session-start" -- grep must use word boundaries.

#### "Void" -- Clean Replace with Caution
- Cosmic "The Void = Harmony space" is clear replace.
- TypeScript `void` return type appears in some files -- must not be replaced.

## Common Pitfalls

### Pitfall 1: Gate Terminology Confusion
**What goes wrong:** Agents rename functional "Pre-Action Gate" and "Gate 1-14" terminology, breaking the orchestrator's operational vocabulary.
**Why it happens:** D-06 says "Gates -> Validation Layer" which is true for the cosmic entity, not the orchestrator checkpoint system.
**How to avoid:** The planner must explicitly instruct: "Gate as a numbered checkpoint (Gate 1, Gate 2, etc.) and 'Pre-Action Gate' section headers are FUNCTIONAL vocabulary -- do NOT rename these. Only rename cosmic descriptions of gates as an entity type."
**Warning signs:** If a rewritten ORCHESTRATOR.md no longer has "Gate 1", "Gate 2", etc., something went wrong.

### Pitfall 2: Harmony System vs Harmony Concept
**What goes wrong:** All references to "harmony" get replaced, including the technical system (`harmonyDetector.ts`, `harmony:check`, `/api/harmony/health`).
**Why it happens:** "Harmony" is both a cosmic concept AND a real backend service name.
**How to avoid:** This phase updates FRAMEWORK FILES (.md only). Code filenames, API paths, and TypeScript names are out of scope. In .md files, "harmony" as a context name becomes "system-health". "Harmony detection" as a system description gets updated to "system health monitoring". But references to actual code paths (harmonyDetector.ts) stay as-is because those are code references, not cosmic terminology.
**Warning signs:** If contract validation commands (`pnpm run harmony:check`) are renamed in docs but not in code, you get broken instructions.

### Pitfall 3: Broken Cross-References After docs/ Deletion
**What goes wrong:** After deleting docs/, 10+ actionflows files still reference `docs/architecture/CONTRACT_EVOLUTION.md`, `docs/architecture/GATE_LOGGING.md`, etc.
**Why it happens:** Cross-references are not automatically updated when targets are deleted.
**How to avoid:** The research documents every broken reference above. The plan must include a cross-reference cleanup task.
**Warning signs:** Any remaining `docs/architecture/` or `docs/living/` path in actionflows files after docs/ is deleted.

### Pitfall 4: Maintenance Flow Orphaning
**What goes wrong:** Maintenance context is removed from CONTEXTS.md but `flows/maintenance/` directory still exists with 6 flow definitions, and FLOWS.md still has a `## maintenance` section.
**Why it happens:** Removing a context from routing without reassigning its flows.
**How to avoid:** The plan must explicitly reassign every maintenance flow to its new context AND update the `flows/maintenance/` directory (either move files or update their content to reference new context).
**Warning signs:** If CONTEXTS.md has no maintenance but FLOWS.md still does, routing is broken.

### Pitfall 5: Incomplete Cosmic Term Grep
**What goes wrong:** After all changes, a stale "living universe" or "cosmic map" reference survives in a template or onboarding module.
**Why it happens:** The grep pattern didn't include all variants, or the replacement was done per-file instead of exhaustively.
**How to avoid:** Use a comprehensive verification grep at the end covering: `universe`, `cosmic`, `living universe`, `big bang`, `hearth`, `observatory`, `cartographer`, `tribunal`, `loom`, `war table`, `vault`, `guardian`, `spy glass`, `Stars` (capitalized, as cosmic entity), `Sparks`, `Moons`, `The Void`, `cosmic map`, `physics` (in cosmic context).
**Warning signs:** Any hit from the verification grep means the phase is incomplete.

### Pitfall 6: CLAUDE.md Inconsistency
**What goes wrong:** `.claude/CLAUDE.md` (inside actionflows) and `.claude/actionflows/project.config.md` have the same cosmic project description. If only one gets updated, agents read inconsistent descriptions.
**Why it happens:** The description block is duplicated between these files.
**How to avoid:** Update BOTH files using the same source text from `.planning/PROJECT.md`.
**Warning signs:** Compare the description blocks in both files after changes.
