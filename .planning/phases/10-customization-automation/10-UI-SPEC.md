# Phase 10 -- UI Design Contract

> Visual and interaction contract for customization & automation surfaces. Generated by gsd-ui-researcher.
>
> Phase 10 delivers five additive UI surfaces on top of the Phase 3 design system and Phase 9 workbench patterns: (1) self-healing approval checkpoints rendered inline in chat, (2) a per-workbench scheduled-tasks manager, (3) a per-workbench skills form, (4) custom workbench creation and sidebar registration, and (5) session forking with branch switching. All five surfaces compose exclusively from existing Phase 3 primitives (Button, Card, Dialog, Tabs, Input, Select, Badge, Tooltip, DropdownMenu) and reuse Phase 9 shared components (WorkbenchGreeting, ContentList, StatCard, FlowCard, FlowBrowser). No new primitive components are introduced — only domain-scoped compositions. No new dependencies beyond Croner 10.0.1 (backend, no UI impact).

| Property | Value |
|----------|-------|
| Tool | none (custom Phase 3 library, shadcn patterns borrowed) |

## Data Flow

### Healing Approval Flow

```
Backend validation detects runtime error
  -> HealingService.onRuntimeError(err, ctx)
  -> HealingQuotaTracker.getTodayQuota checks daily limit
  -> If within quota: ApprovalService.createRequest({ action: 'heal_runtime_error' })
  -> WebSocket broadcast: { type: 'healing:approval_requested', approval, error, proposedFix }
  -> Frontend healingStore receives event, pushes to pendingApprovalId
  -> Chat panel renders HealingApprovalCard inline
  -> User clicks Approve | Investigate
  -> POST /api/approvals/{id}/resolve { decision }
  -> Backend HealingService runs fix (if approved) or stops flow (if declined)
  -> WebSocket broadcast: { type: 'healing:resolved', attemptId, status }
  -> Frontend toast + card removal
```

### Scheduled Task Creation Flow

```
User clicks "Create Task"
  -> ScheduledTaskDialog opens
  -> User types cron expression
  -> Frontend calls Croner.parse() live for next-run preview
  -> User clicks "Save Task"
  -> POST /api/scheduled-tasks/{workbenchId} { name, description, cronExpression, target }
  -> Backend ScheduledTaskService.registerTask creates Croner instance
  -> Backend writes to Storage: `schedule:{workbenchId}:{taskId}`
  -> Response: { task with nextRun populated }
  -> Frontend scheduleStore adds to tasks map
  -> ScheduledTaskRow renders with next-run timestamp
```

### Custom Workbench Creation Flow

```
User clicks "Create Workbench"
  -> CustomWorkbenchDialog opens
  -> User fills form (name, icon, greeting, tone, system prompt)
  -> User clicks "Save Workbench"
  -> POST /api/custom-workbenches { name, iconName, greeting, tone, systemPromptSnippet }
  -> Backend customWorkbenchService writes to Storage: `customWorkbench:{id}`
  -> Backend registers WorkbenchMeta, makes workbench ID valid for subsequent session/flow/skill APIs
  -> Response: { workbench }
  -> Frontend customWorkbenchStore adds to workbenches list
  -> Sidebar re-renders, new custom workbench item appears below divider
  -> Clicking new item activates its empty session (or creates one via SessionManager)
```

### Session Forking Flow

```
User hovers chat message, clicks ForkButton
  -> ForkDialog opens
  -> User enters description, clicks "Create Fork"
  -> POST /api/forks { parentSessionId, workbenchId, description, forkPointMessageId }
  -> Backend ForkMetadataService.createFork:
      - Calls existing SessionManager.forkSession() (uses Agent SDK forkSession)
      - Persists ForkMetadata to Storage
  -> Response: { forkMetadata with new forkSessionId }
  -> Frontend forkStore adds to forksByParent map
  -> ForkSwitcher appears above chat (or new tab added)
  -> forkStore.switchBranch(parentSessionId, forkSessionId) activates new branch
  -> Chat panel re-renders with new session's messages (empty initially, then copies from fork)
```

### Fork Merge Flow

```
User clicks DropdownMenu "Merge Fork"
  -> MergeDialog opens with RadioGroup
  -> User selects resolution strategy
  -> (if manual) types merged content
  -> User clicks "Merge"
  -> POST /api/forks/{forkId}/merge { resolution, manualContent? }
  -> Backend ForkMetadataService.mergeFork:
      - Creates new session via SessionManager with merged context
      - Marks original fork status='merged'
  -> Response: { mergedSessionId }
  -> Frontend forkStore marks fork as merged
  -> ForkSwitcher removes merged fork tab
  -> Chat switches to merged session, shows "Merge commit" marker
```

## Design Decisions Rationale

| Decision | Rationale |
|----------|-----------|
| Place all 4 Phase 10 sub-panels under Settings (not scattered per-workbench) | RESEARCH.md explicitly places all panels in `workbenches/settings/`. Scattering skills/schedules to each of 7+ workbenches would duplicate forms. Single Settings surface, scoped by activeWorkbench, keeps CRUD centralized. |
| No sidebar "Create Workbench" button (moved to Settings tab) | Avoids sidebar overflow with unlimited custom workbenches. Phase 4 sidebar already has 7 items — adding a global form button would mix navigation with CRUD concerns. |
| Custom workbenches visually identical to defaults in sidebar (only "Custom" divider label distinguishes) | Specifics.md hint "visually distinct from defaults" is satisfied by the group divider + label, not by per-item color changes. Custom workbenches should feel first-class, not second-tier. |
| HealingApprovalCard as full Card with warning-colored left border (not compact inline alert) | Matches Phase 8 ApprovalCard precedent. Healing is a decision gate requiring two buttons + quota counter — compact alert cannot accommodate this. |
| Fork switcher as Tabs (not tree view) | D-12 explicitly chose "listed as tabs/separate views". Radix Tabs gives free keyboard nav + ARIA. Tree visualization is explicitly deferred. |
| Fork description required | D-14 locked choice ("required fork description"). Dialog blocks submission until field is non-empty. |
| Merge resolution as RadioGroup (not visual merge tool) | D-13 discretion: "simple text choice (theirs/parent/manual)". Visual merge UI is deferred. RadioGroup is the canonical Phase 3 primitive for mutually-exclusive choices. |
| Delete confirmation dialog focuses Cancel button by default | Accessibility best practice for destructive actions — safer default focus target. |
| Default workbenches cannot be deleted (no delete button rendered) | D-11 locked: "cannot modify the 7 default workbenches". Enforce at UI layer (no button) AND service layer (REST rejects). |
| Icons picked from lucide-react curated subset | Phase 3 icon library commitment. Curating ~30 options prevents icon-picker bloat while giving users meaningful choice. |
