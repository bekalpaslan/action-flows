# Phase 9 -- UI Design Contract

> Visual and interaction contract for workbench content pages and flow management. Generated by gsd-ui-researcher.
>
> Phase 9 replaces 7 placeholder workbench pages with domain-specific content. Each workbench gets a tailored layout, personality-driven greeting, and a flow browser showing registered flows as card grids. A flow composition UI allows users to build new flows by selecting and reordering actions from the catalog. All visual values inherit from the Phase 3 design system (theme.css tokens, component library). No new dependencies are needed beyond what is already installed. All components compose exclusively from existing primitives (Card, Badge, Button, Input, Select, Tabs, Dialog).

## Interaction Contracts

### Flow Card Interactions

| Interaction | Behavior |
|-------------|----------|
| Hover on flow card | border transitions from border to border-strong (Card interactive variant) |
| Click "Run" button | Sends flow name as chat message to workbench agent. Button shows loading state (disabled + spinner) for 2 seconds. Toast notification: "Flow '{name}' started." |
| Run button disabled state | Disabled when workbench agent session is disconnected |

### Flow Composition Interactions

| Interaction | Behavior |
|-------------|----------|
| Click "New Flow" | Opens FlowComposer Dialog |
| Search actions | Filters action list in real-time as user types |
| Click action item | Toggles selection (adds/removes from chain order list) |
| Drag action in chain list | Reorders the action within the chain sequence |
| Click "Save Flow" | Validates flow name (non-empty, kebab-case), saves to flow registry, closes dialog, shows success toast |
| Click "Cancel" | Closes dialog without saving, no confirmation needed |
| Empty flow name submission | Input shows error variant, focus stays on input |

### Content List Interactions

| Interaction | Behavior |
|-------------|----------|
| Hover on list item | Background transitions to surface-2 |
| Click list item | No action in v1 -- items are informational. Future: navigate to detail view. |
| Scroll | Standard overflow-y-auto within tab content area |

### Tab Interactions

Standard Radix Tabs behavior. Keyboard: Arrow Left/Right to switch tabs. Active tab shows accent bottom border (existing TabsTrigger styles).

### Greeting Card Interactions

Non-interactive. Display-only card showing agent personality greeting. No click, hover, or focus behaviors.

| Requirement | Implementation |
|-------------|---------------|

| Run button | Button component handles focus-visible natively. Accessible label: "Run {flow name}" via aria-label |

## Data Flow

### Flow Loading

```
App mount
  -> flowStore.loadFlows()
    -> GET /api/flows (backend parses FLOWS.md)
    -> flowStore.flows = parsed result
    -> Each workbench page reads flowStore.getFlowsByContext(workbenchId)
```

### Flow Execution

```
User clicks "Run" on FlowCard
  -> FlowCard calls chat panel sendMessage(`/run ${flowName}`)
  -> Chat panel delivers message to workbench agent
  -> Agent compiles and executes chain (existing Phase 7 infrastructure)
  -> Pipeline updates via WebSocket (existing Phase 5 infrastructure)
```

### Flow Composition

```
User clicks "New Flow"
  -> FlowComposer dialog opens
  -> User selects actions from parsed ACTIONS.md catalog
  -> User reorders via drag-and-drop
  -> User names the flow
  -> "Save Flow" -> POST /api/flows { name, context, chain }
  -> Backend writes to FLOWS.md or dedicated storage
  -> flowStore.addFlow() updates local state
  -> Dialog closes, flow appears in FlowBrowser
```
