Create the hook-based design system enforcement layer -- PreToolUse and PostToolUse hook scripts that validate agent file edits against component library rules using regex pattern matching. Includes unit tests for the design rules module.

- `no-raw-hex`: regex `/(?:color|background(?:-color)?|border(?:-color)?|fill|stroke|outline-color)\s*:\s*#[0-9a-fA-F]{3,8}\b/g` with message "Raw hex color detected. Use var(--color-*) design tokens."
- `no-raw-color-fn`: regex `/(?:color|background(?:-color)?|border(?:-color)?|fill|stroke)\s*:\s*(?:rgb|rgba|hsl|hsla)\s*\(/g` with message "Raw color function detected. Use var(--color-*) design tokens."

Export `WARNING_PATTERNS` array (warned in PostToolUse):
- `hex-outside-tokens`: regex `/['"]#[0-9a-fA-F]{3,8}['"]/g` with message "Hex color string found. Consider using a design token reference."

Export `VALIDATED_EXTENSIONS`: `['.tsx', '.css']` per D-04.

Export `SKIP_PATTERNS`: regexes for `.test.`, `.spec.`, `.config.`, `design-tokens`, `theme\.css`, `__tests__`, `__mocks__`, `node_modules`.

Each pattern entry has shape: `{ rule: string; pattern: RegExp; message: string; }`.
Export the type `DesignRule` for this shape.

**1. Create `packages/hooks/src/afw-design-validate-pre.ts`** (PreToolUse hook per D-02):

Follow the established hook pattern from `afw-control-check.ts`:
- Read JSON from stdin: `const inputData = fs.readFileSync(0, 'utf-8');`
- Parse as `PreToolUseInput` with fields: `session_id: string`, `tool_name: string`, `tool_input: { file_path?: string; content?: string; new_string?: string; edits?: Array<{ new_string: string }> }`
- Normalize `file_path` with `path.normalize(filePath).replace(/\\/g, '/')` (Windows path fix per RESEARCH Pitfall 4)
- Check extension with `path.extname()` against `VALIDATED_EXTENSIONS` (D-04: only .tsx and .css)
- Check against `SKIP_PATTERNS` (test files, config, design-tokens.css, theme.css)
- Extract content to check: if `tool_name === 'Write'` use `tool_input.content`; if `'Edit'` use `tool_input.new_string`; if `'MultiEdit'` iterate `tool_input.edits` array and join `.new_string` values (RESEARCH Pitfall 3)
- Check `CRITICAL_PATTERNS` against content. Reset regex lastIndex before each match (global flag).
- If violations found:
  - Write to stderr: `"Design System Violation (BLOCKED):\n"` + joined violation messages + `"\n\nFix: Use var(--color-*) tokens from design-tokens.css instead of raw values.\nReference: packages/app/src/styles/theme.css"`
  - Call `reportViolation(normalizedPath, violationMessages, sessionId)` fire-and-forget (no await)
  - `process.exit(2)` to block the tool call
- If no violations: `process.exit(0)` to allow
- Wrap in `main().catch(() => process.exit(0))` -- silent failure = allow (per RESEARCH)

**2. Create `packages/hooks/src/afw-design-validate-post.ts`** (PostToolUse hook per NEURAL-03/06):

Follow PostToolUse output format from RESEARCH Pattern 1:
- Read JSON from stdin (same as PreToolUse)
- Same file_path normalization, extension check, skip patterns
- Extract content the same way as PreToolUse (Write/Edit/MultiEdit handling)
- Check `WARNING_PATTERNS` against content
- Also check `CRITICAL_PATTERNS` as defense-in-depth (RESEARCH Pitfall 6) -- report at warning level, not block
- If issues found:
  - Output JSON to stdout with `hookSpecificOutput.additionalContext` containing the warning messages (NEURAL-06: semantic compliance via additionalContext)
  - Call `reportViolation(normalizedPath, warningMessages, sessionId)` fire-and-forget with severity 'warning'
  - `process.exit(0)` (PostToolUse does NOT block, only advises)
- If no issues: `process.exit(0)` silently
