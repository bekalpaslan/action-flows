# Phase 8 -- UI Design Contract

> Visual and interaction contract for the neural validation and safety UI surfaces. Generated by gsd-ui-researcher.
>
> Phase 8 is primarily a backend/hooks phase (PreToolUse/PostToolUse design system enforcement, /btw signaling). The frontend surfaces are extensions of existing components from Phase 5 (pipeline) and Phase 7 (chat). Three new UI surfaces are introduced:
>
> 1. **Checkpoint timeline markers** on pipeline step nodes -- visual indicators with one-click revert (SAFETY-01, D-08/D-09/D-10)
> 2. **Approval gate cards** in the chat panel -- inline approve/deny for destructive operations (SAFETY-02/03/04, D-11/D-12/D-13)
> 3. **Violation signal toasts** -- severity-colored notifications for design system violations (NEURAL-04, D-05/D-06/D-07)
>
> No new pages, no new panels. All visual values inherit from the Phase 3 design system (theme.css tokens, component library). New components compose exclusively from existing primitives (Badge, Button, Card, StatusDot, Toast via sonner).

## Interaction Contracts

### 1. Checkpoint Timeline (SAFETY-01, D-08/D-09/D-10)

**Where:** Pipeline panel, below each StepNode that has a completed checkpoint.

**Visual:** A small 8x8 circle marker positioned below the step node. Default state is surface-3 background with border. On hover, the marker transitions to accent color and a tooltip appears.

**Tooltip content:**
- Line 1: Commit message (truncated to 60 chars), text-caption font-semibold
- Line 2: Timestamp in relative format ("2m ago"), text-caption text-text-muted font-mono
- Line 3: "Revert to this point" button, outline variant, sm size

**Revert flow:**
1. User hovers checkpoint marker -- tooltip appears (150ms delay)
2. User clicks "Revert to this point" -- confirmation Dialog opens
3. Dialog shows: "Revert to checkpoint?" heading, commit message, "This will create a new commit that undoes all changes after this point." body, two buttons: "Revert changes" (destructive Button variant) and "Keep changes" (ghost Button variant)
4. On confirm -- backend executes `git revert`, toast confirms success or shows error
5. Pipeline nodes after the reverted checkpoint visually reset to pending status

**States:**
- Default: surface-3 dot, no interaction
- Hover: accent dot, tooltip visible
- Reverting: spinner replaces dot, tooltip shows "Reverting..."
- Reverted: success dot briefly (2s), then returns to default

### 2. Approval Gate Cards (SAFETY-02/03/04, D-11/D-12/D-13)

**Where:** Chat panel, inline as a special message type. Same position as AskUserQuestion cards.

**Visual:** A card with border-l-2 border-warning (orange left accent), surface-2 background, 12px padding. Follows the AskUserRenderer pattern exactly but with approve/deny semantics instead of question/answer.

**Card content:**
- Row 1: Warning icon (ShieldAlert from lucide) + "Approval Required" heading, text-body font-semibold
- Row 2: Description of the action ("Agent wants to delete 3 files"), text-caption text-text-dim
- Row 3: File list (if applicable), each file path on its own line, text-caption font-mono
- Row 4: Workbench name + autonomy level badge, text-caption
- Row 5: Button row -- "Approve action" (primary Button, sm) + "Deny action" (ghost Button, sm)

**Interaction flow:**
1. Backend sends approval_request via WebSocket
2. Card renders in chat panel at current message position
3. User clicks "Approve action" or "Deny action"
4. Button row replaced with result badge: "Approved" (success Badge) or "Denied" (error Badge)
5. Card becomes non-interactive (opacity-75, cursor-default) -- same pattern as submitted AskUserRenderer

**Timeout behavior (Claude's Discretion):**
- Gate auto-denies after 120 seconds if no user response
- A progress indicator (thin bar at top of card) depletes over 120s using accent color
- At 30s remaining, bar color transitions to warning
- At 10s remaining, bar color transitions to destructive
- On timeout, "Timed out" badge (default Badge variant) replaces buttons

**States:**
- Pending: Full card with approve/deny buttons, progress bar counting down
- Approved: Buttons replaced with "Approved" success badge, card at 75% opacity
- Denied: Buttons replaced with "Denied" error badge, card at 75% opacity
- Timed out: Buttons replaced with "Timed out" default badge, card at 75% opacity

### 3. Violation Signal Toasts (NEURAL-04, D-05/D-06/D-07)

**Where:** Top-right corner of the app, via sonner toast system (already installed from Phase 6).

**Visual:** Custom toast component with severity-colored left border. Uses existing sonner positioning and animation.

**Toast content:**
- Left border: 3px solid, color per severity (destructive/warning/info)
- Row 1: Severity badge (Badge component, variant matching severity) + violation rule name, text-body font-semibold
- Row 2: Description of what was detected ("Raw hex color #fff found"), text-caption text-text-dim
- Row 3: File path + line number, text-caption font-mono

**Duration per severity:**
- Critical: Persistent (user must dismiss), auto-dismiss after agent fixes the violation
- Warning: 8 seconds
- Info: 5 seconds

**Stacking:** Maximum 3 visible toasts. Additional violations queue and appear as earlier ones dismiss. A counter badge appears on the 3rd toast if more are queued ("+ 2 more").

**States:**
- Visible: Full toast with content
- Dismissed: Slide-out animation (sonner default)
- Auto-fixed: Brief green flash (success border) before slide-out, indicating agent resolved the issue

### 4. Autonomy Level Settings (SAFETY-05, D-12)

**Where:** Settings workbench page, as a section within the existing SettingsPage.tsx.

**Visual:** A card (surface-2 background, border, radius-md) containing a list of workbenches with their autonomy level controls.

**Card content:**
- Header: "Workbench Autonomy Levels" heading, text-heading font-semibold
- Subheader: "Control what actions require human approval per workbench", text-caption text-text-dim
- List: One row per workbench (7 rows), each containing:
  - Workbench icon (lucide, from sidebar) + name, text-body
  - Select component (existing Phase 3 Select) with 3 options:
    - "Full" -- auto-approve everything
    - "Supervised" -- approve destructive only
    - "Restricted" -- approve all edits

**Default autonomy levels (Claude's Discretion, per D-12):**

| Workbench | Default Level | Rationale |
|-----------|--------------|-----------|
| Work | Supervised | Primary workspace, destructive-only gates |
| Explore | Supervised | Read-heavy, but file edits should be gated |
| Review | Restricted | Auditing context, all edits require approval |
| PM | Supervised | Planning workspace, destructive-only gates |
| Settings | Full | System config, user is explicitly configuring |
| Archive | Restricted | Historical data, protect from accidental changes |
| Studio | Full | Preview/experiment space, maximum autonomy |

**Interaction:** Selecting a new autonomy level updates immediately (optimistic UI). A brief success toast confirms "Work autonomy set to Supervised." No save button needed -- changes persist via backend API call on selection change.

## Accessibility

| Requirement | Implementation |
|-------------|---------------|
| Checkpoint markers | role="button", aria-label="Checkpoint: {commit message}", keyboard focusable (Tab), Enter/Space to open tooltip |
| Revert confirmation dialog | Focus trapped in dialog, Escape to close, autofocus on "Keep changes" button (safe default) |
| Approval gate cards | role="group", aria-label="Approval request: {action description}", "Approve action"/"Deny action" buttons keyboard accessible |
| Violation toasts | role="alert" for critical severity, role="status" for warning/info, aria-live="assertive" for critical, "polite" for others |
| Autonomy settings | Each Select has aria-label="{workbench} autonomy level", fieldset grouping with legend |
| Timeout progress bar | aria-hidden="true" (decorative), actual timeout communicated via aria-live region when < 30s |
| Reduced motion | All animations respect prefers-reduced-motion via existing --duration-* token overrides (already 0ms in reduced motion) |

## New Shared Types

Types to add in `packages/app/src/lib/chat-types.ts`:

```typescript
export type ApprovalStatus = 'pending' | 'approved' | 'denied' | 'timed_out';

export interface ApprovalRequest {
  id: string;
  action: string;           // "delete_files" | "force_push" | "drop_table" | etc.
  description: string;      // Human-readable: "Delete 3 files from packages/app/"
  files?: string[];          // Affected file paths (if applicable)
  workbenchId: string;
  autonomyLevel: 'full' | 'supervised' | 'restricted';
  status: ApprovalStatus;
  expiresAt: string;         // ISO timestamp, 120s from creation
  resolvedAt?: string;       // ISO timestamp of approval/denial/timeout
}
```

Types to add in `packages/app/src/lib/pipeline-types.ts`:

```typescript
export interface CheckpointData {
  commitHash: string;        // Short hash (7 chars)
  commitMessage: string;     // First line of commit message
  timestamp: string;         // ISO timestamp
  filesChanged: number;      // Count of files in this commit
}
```

Extend `StepNodeData`:

